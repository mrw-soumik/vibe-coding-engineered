# Sample Jira Backlog

## MVP-001: Define restaurant FAQ MVP scope

As a founder, I want to define the MVP scope so that the team avoids overbuilding.

Acceptance criteria:

- In-scope items are documented.
- Out-of-scope items are documented.
- Risky automation boundaries are documented.

## MVP-002: Build founder notes input workflow

As a founder, I want to paste customer notes so that the system can analyze them.

Acceptance criteria:

- Text input is accepted.
- Empty input is handled.
- Long input is handled.

## MVP-003: Generate MVP analysis output

As a founder, I want clear analysis so that I can understand the problem and MVP.

Acceptance criteria:

- Problem is extracted.
- Target user is identified.
- Pain points, risks, and assumptions are listed.

## MVP-004: Generate Jira-style execution tickets

As a founder, I want buildable tickets so developers can execute the MVP.

Acceptance criteria:

- Tickets have user stories.
- Tickets have acceptance criteria.
- Tickets have branch names.

## MVP-005: Add evaluation metrics

As a founder, I want quality checks so that the MVP is not just a nice-looking demo.

Acceptance criteria:

- Content quality is scored.
- Domain fit is scored.
- System reliability is scored.

## MVP-006: Create final founder summary

As a founder, I want a clear delivery summary so I can share progress with my team, mentors, or investors.

Acceptance criteria:

- The summary states the problem and the selected MVP.
- The summary lists what was built and tested.
- The summary includes recommended next steps.
